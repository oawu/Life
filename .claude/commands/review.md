# Code Review

審查本次對話中所做的所有程式碼變更，產出結構化的 review 報告。

## 步驟

### 1. 收集變更範圍

透過 `git log` 和 `git diff` 確認本次對話產生的所有 commit 與變更檔案。

### 2. 逐檔審查

**依重要性排序**，依序檢查每個變更檔案：

#### ① Bug 與正確性（最高優先）
- 語言層級錯誤（語法、型別、API 誤用）
- 邏輯錯誤、邊界條件、null/undefined 風險
- 非同步操作是否正確處理（race condition、error handling）
- 事件監聽器是否正確清理（memory leak）
- DOM 操作時序是否安全

#### ② 功能需求
- 是否達成用戶要求的功能目標
- 有無遺漏的邊界情境或未處理的狀態

#### ③ 流程與邏輯衝突
- 與既有功能是否衝突（事件冒泡、全域狀態、CSS 權重）
- 元件間的依賴關係是否正確
- API 請求/回應格式是否一致
- 引用完整性（重命名/搬移時）：`window.Core` 註冊、`__tmpl__` 路徑、asset 路徑、文件同步

#### ④ 開發規範
- `.claude/rules/` 下所有規範（js、php、scss、el3、git、design）
- **SCSS 必須先讀 el3 比對巢狀結構**，再檢查屬性順序
- 命名慣例（私有 `_` 前綴、setup 區塊順序、`=>` 對齊）
- 程式碼風格一致性（與同檔案既有程式碼比對）

#### ⑤ 優化建議（非必要）
- 可簡化或合併的程式碼
- 效能改善空間（不必要的 re-render、重複查詢）
- 可讀性改善

### 3. 產出報告

以表格形式整理每個檢查維度的結果：

```
## Review 結果

### [功能名稱 1]

| 檢查項目 | 結果 |
|----------|------|
| ... | ... |

### [功能名稱 2]
...

## 結論

- 是否有 bug 需要修復
- 是否有規範違反需要調整
- 優化建議（非必要，列出供參考）
```

### 4. 修復問題（若有）

若發現需要修復的問題：
1. 說明問題與修復方案
2. 詢問是否要立即修復
3. 修復後重新驗證

## 注意事項

- 審查範圍僅限本次對話的變更，不擴大到未修改的程式碼
- 歷史報告檔案（docs/report/YYYY-MM-DD.md）為歷史快照，不需要更新
- 優化建議只提非必要的改善，不強制執行
- 規範以 `.claude/rules/` 為準，不自行發明規則
